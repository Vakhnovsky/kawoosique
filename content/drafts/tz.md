Привет! Перед тобой исходный код Универсального Telegram-бота для публикации Rich-Markdown постов в каналы. 
Бот написан на Python с использованием асинхронного фреймворка aiogram 3.29+ и SQLite в качестве БД.
Проект находится на этапе локальной разработки и запускается только на моем ПК внутри виртуального окружения (venv).

ПРАВИЛА БЕЗОПАСНОСТИ ДЛЯ ТЕБЯ (AI-агент):
1. Никогда не хардкодь токены бота (BOT_TOKEN) или ID администраторов в коде. Все настройки должны строго импортироваться из переменных окружения через python-dotenv.
2. Файл конфигурации окружения должен называться строго `.env`. Если тебе нужно показать пример настроек окружения, создай файл `.env.example` с пустыми значениями.
3. Не изменяй структуру таблиц в базе данных SQLite без явного предупреждения, чтобы не затереть тестовых пользователей.

ТЕКУЩАЯ ЗАДАЧА:
Реализация и отладка 3-х сценариев публикации Markdown в асинхронном Telegram-боте (aiogram 3.29+, Telegram Bot API 10.1+).

Контекст: Блог запускается исключительно ЛОКАЛЬНО на ПК разработчика. Бот должен поддерживать форматирование InputRichMessage (нативный Rich-Format) и обеспечивать предпросмотр (Preview) перед отправкой в канал.

Тебе необходимо реализовать/доработать обработку контента по трем сценариям (сохранив при этом уже созданный функционал и логику работы бота):

---

### СЦЕНАРИЙ №1: Сырой текст с веб-ссылками изображения
- Пользователь отправляет боту напрямую сообщение с разметкой.
- Картинки встроены через HTML-теги: <img src="https://host.com/photo.jpg"/> или стандартные веб-линки Markdown: ![img](https://host.com/photo.jpg).
- Ожидаемое поведение: Бот нативно парсит форматирование, оставляет внешние ссылки нетронутыми, генерирует корректный инлайн-объект превью и выводит под ним кнопку "[Опубликовать]".

### СЦЕНАРИЙ №2: Отправка файла .md с веб-ссылками изображения
- Пользователь отправляет в чат боту документ с расширением `.md` (или `.txt`).
- Внутри файла находится Markdown-текст, где картинки также указаны в виде внешних URL-адресов.
- Ожидаемое поведение: Бот считывает текст из файла, обрабатывает его по логике Сценария №1 и выдает превью в чат с кнопкой публикации.

### СЦЕНАРИЙ №3: Локальная публикация поста Hugo / Page Bundles (КРИТИЧЕСКАЯ ФИЧА)
Пользователь передает боту файл `index.md` готовой статьи для блога Hugo. Файл содержит блок метаданных Front Matter (YAML) и ссылки на локальные изображения, лежащие в ТОЙ ЖЕ папке на жестком диске.

Пример структуры Front Matter в файле:
---
title: Ещё одна и одна попытка
date: 2026-06-27
tg_mode: rich_only
tags: [start, welcome]
cover:
  image: zasedanie.jpg
  relative: true
---

Внутри тела статьи картинка прописана локально: ![картинка](tastefully.jpg) или в синтаксисе Obsidian: ![[tastefully.jpg]].

Требования к реализации Сценария №3:
1. Определение путей (Два гибких режима):
   - Режим А: Бот автоматически определяет родительскую папку переданного файла index.md на ПК (так как запуск локальный) и считает её корневой рабочей директорией для поиска картинок.
   - Режим Б (Резервный): Бот берет базовый путь из переменной окружения `.env` (например, HUGO_POSTS_DIR=e:\GitHub\KAWOOSIQUE\content\posts\), вытягивает из Front Matter кастомный параметр папки (например, folder: 20260627-last-test) и собирает абсолютный путь к файлам на диске.
2. Парсинг Front Matter: Скрипт должен корректно отсекать YAML-блок от основного текста статьи. Из метаданных нужно вытянуть заголовок (`title`), теги (`tags`), и имя главного изображения обложки (`cover.image`).
3. Загрузка локальных медиафайлов (Главная проблема проекта):
   - Бот должен найти на диске файл обложки (zasedanie.jpg) и файлы картинок из текста (tastefully.jpg).
   - Скрипт должен прочитать эти файлы в байты и загрузить их на Telegraph API (POST /upload) или альтернативный стабильный хостинг (например, Catbox/Litterbox multipart-формой), получив в ответ валидные веб-ссылки (URL).
   - В итоговом тексте сообщения бот должен автоматически подменить локальные имена картинок на полученные URL.
4. Вывод: Из обложки (`cover_url`) и обработанного текста формируется InputRichMessage. Бот присылает пользователю превью поста.
